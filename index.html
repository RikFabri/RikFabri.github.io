Chip-8

I wrote this C++ CHIP-8 VM/interpreter because I wanted to experiment with cmake, QT and learn about emulator development. This project combines it all!
The project is fully functional, except for sound. However, I'm doing my graduation work on gameboy sound and will probably add sound support here afterwards.


Emulation

Opcodes are the heart of any emulator (or cpu). For Chip-8 each opcode is 2 bytes (big-endian) consisting of bits to identify the opcode and bits that contain data. For example, "00EE" returns from a subroutine and "2NNN" calls a subroutine at whatever value "NNN" has. The following code is used to identify an opcode and know which function it relates to.

Sprites

Chip-8 has an interesting way of handling sprites and collisions. Drawing a sprite uses opcode "DXYN" which provides us with both the registers for the x and y coordinates and the height of the sprite (n). To actually draw the sprite, we read a byte for each row in the sprite (n) at the memory address stored in I (this is a special 16bit register meant to store memory addresses). Each bit represents whether or not a pixel is white. The pixel is then xor'ed onto the display, if an older pixel is unset, a collision flag is set for later use. This way, we can draw sprites that are 8 pixels wide (1 byte) and n-rows long.

NOX

Description

Nox is a 2,5D atmospheric puzzle platform game set in an 80's arcade. Made for the DAE course "game projects". Most of the codebase is written in Unreal C++, although we also used blueprints when that made more sense. The entire project was under perforce version control.


Team

We were a team of 2 programmers and 4 artists (Both me and the other programmer had no prior unreal C++ experience). Everyone put in around 80-100 hours. I myself did most of the gameplay programming, including:


  • Pushing/Pulling
  • Spiderwebs that lock objects
  • Climbing
  • Ziplining
  • Rope splinemeshes
  • Collectables base
  • General purpose levers, easily extensible in blueprints for our level designers
  • Bossfight (generating leg movement through IK, spider-pushable and spider-player interactions)
  • ...

The Crunch

Game Jam

I made this game with a good friend for the Ludum Dare 42 game jam. This was before I started studying at DAE and even though I knew a decent amount about C++, the good practices were definitely not there.


Mistakes

There's so much wrong with the codebase that I can't possibly give you focused code snippets. Restarting the game was an afterthought I fixed with a goto, causing the biggest memory leak imaginable. I knew about constructors, but not about the initializer list and didn't use the destructor. So goodbye RAII principles and hello even more memory leaks! The Spaceship's render method wasn't const and has a shocking amount of gameplay logic in there.
The list goes on, really.
(You can find the code via our LDJam page, but I won't link to it directly to prevent people from opening it without reading this.)


Conclusion

I guess the reason this project is still on my portfolio is simply that it was my first "real" game, we did a pretty good game jam even though our skillset was lacking to say the least.
It's also an indicator on how far I've come since then and a reminder of why I got into videogames.

Tug of TAR [WIP]

Tug of tar is a work-in-progress game where people can place space stations in a shared AR-space to have epic battles over precious oil.
We're using Unity, MLAPI, Azure spatial anchors, MRTK and version control everything with perforce.


The team

We're a team of 3 programmers and 2 artists. Currently, my main responsibility (next to game design which we all do together) is everything networking related. We're using Unity's new MLAPI for this.
